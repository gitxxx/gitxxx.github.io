同步架构

涉及如下三个组件：

  • Kafka Connect
  • Schema Registry
  • Apache Kafka

Apache kafka由Linkedin开发并开源。Kafka Connect和Schema Registry由Confluent公司开发并开源


组件解释：
Kafka Connect
Connector分为如下两类：

  • 从source系统中同步数据到Kafka
  • 从Kafka中获取数据同步到sink系统中。

文档地址：Connectors

Schema Registry
Schema Registry为metadata提供了一个服务层。它提供了一个用于storing和retrieving Avro架构的RESTful接口。 它存储所有schema的版本化历史记录，提供多个兼容性设置，并允许根据配置的兼容性设置演进schemas。
解耦生产端和消费端，schema如有修改，消费端可直接获取到。当下游有多个使用方时，使用Schema Registry服务效果更佳明显。
文档地址：schema registry

  • 基于CDC的方式
  • 基于触发器+JDBC的方式

Kafka 消息格式选择

消息格式选择时，主要考虑如下两个因素：

  • 方便
  • 效率（时间、空间）

Kafka提供两种序列化方式：

  • String：org.apache.kafka.common.serialization.StringSerializer
  • ByteArray：org.apache.kafka.common.serialization.ByteArraySerializer

StringSerializer主要是简单消息或者JSON，或者XML，XML用的比较少。
ByteArraySerializer可以使用官方推荐的Avro，或者自定义的序列化Bean。

JSON

优点：

  • 数据格式完全不固定，而且大量的文字性的消息，它足够灵活，又没有XML那么罗嗦。

缺点：

  • 每一个record都会重复每个字段的名称，这使得JSON在大量使用时效率低下。
  • JSON的字段没有类型，不够直白。

Avro

confluent官方推荐的消息格式，avro的优点，

  1. 它与JSON有直接映射关系
  2. 它有一个非常紧凑的格式。大量的JSON，每一个记录都会重复每个字段的名称，这使得JSON在大量使用时效率低下。
  3. 它非常fast,二进制消息,性能好/效率高
  4. 它对各种编程语言都有很好的bindings，所以您可以生成使对event data的处理更容易的Java对象，但不需要生成代码，因此可以为任何数据流编写一般工具。
  5. 它具有在纯JSON中定义的丰富，可扩展的schema语言
  6. 它具有最佳的兼容性概念，可以随着时间的推移发展您的数据。

详情可以看avro-kafka-data这篇文章。
